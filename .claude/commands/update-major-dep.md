pnpm outdatedを実行し、メジャーバージョンに差分がある依存パッケージの中から最も安全なものを1つ選んでアップデートしてください。

## 対象

`$ARGUMENTS` でパッケージ名が指定された場合はそのパッケージを対象にする。
指定がない場合は `pnpm outdated` の結果からメジャーバージョン差分があるものを全て洗い出し、その中から選ぶ。

## 手順

### 1. 差分の洗い出し

- `pnpm outdated` を実行し、Current/Latestでメジャーバージョンが異なるパッケージを一覧化する

### 2. 安全性の評価

候補それぞれについて以下を確認し、最もリスクが低いものを選ぶ。

- **型定義のみか実行時コードか**: `@types/*` はビルド時の型チェックにしか影響せず実行時の挙動を変えないため、相対的に安全
- **実際にAPIを使っている範囲**: 該当パッケージのAPIを直接使っている箇所を `grep` 等で確認し、影響範囲を把握する
- **Latestが安定版か**: 例えば `@types/node` はNodeの偶数メジャー（LTS）に対応するバージョンを優先し、奇数メジャー（非LTS）の最新は避ける。ローカルの `node -v` や CI（`.github/workflows/*.yml` の `setup-node`）で使われているバージョンと照らし合わせて判断する
- **メジャーリリースの性質**: TypeScriptやESLintのメジャーアップのように大きな破壊的変更を伴うものは、型定義のみの更新より優先度を下げる

### 3. アップデートの実行

- `pnpm add -D <package>@<version>`（`dependencies`側なら `-D` なし）で該当パッケージのみを更新する
- 複数候補があっても一度に1つだけ上げる

### 4. 検証

以下を順に実行し、すべて成功することを確認する。

- `pnpm exec tsc --noEmit`
- `pnpm lint`
- `pnpm build`

`pnpm lint` で `pnpm-lock.yaml` のフォーマット差分が指摘された場合は、ローカルのpnpmバージョン差によるクォートスタイルの揺れであることが多いので `pnpm exec prettier --write pnpm-lock.yaml` を実行してから再度確認する。

## 出力

- 選んだパッケージと選定理由（他の候補と比べてなぜ安全と判断したか）を簡潔に説明する
- 型チェック・lint・buildの検証結果を報告する
- ユーザーの指示がない限りコミットは行わない

## 注意事項

- 一度に複数パッケージを上げない
- Latestが必ずしも「安定した最新」とは限らない（非LTS版など）ため、実際にインストールするバージョンは吟味する
- CIやデプロイ環境のNodeバージョンが分かる場合は、それとの整合性も考慮する
